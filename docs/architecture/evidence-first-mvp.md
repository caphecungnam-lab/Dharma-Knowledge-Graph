# Evidence-First MVP Architecture

This note defines the evidence-first layer for the 21-day MVP. It extends the
existing Dharma Knowledge Graph ontology without replacing the current
`Concept`, `Text`, `Citation`, `Term`, `School`, `Person`, and `Place` model.

## Goal

The MVP should make every concept-level claim traceable to inspectable evidence.
The graph should answer three practical questions before adding APIs,
embeddings, or LLM workflows:

1. Which corpus did this material come from?
2. Which document or source produced this evidence?
3. Which concept is evidenced by the cited passage or note?

## Evidence-First Chain

The core chain is:

```text
Corpus -> Source -> Document -> Evidence -> Concept
                         \          \
                          \          -> Citation
                           -> Work
```

The existing ontology remains useful:

- `Concept` remains the stable doctrinal or philosophical idea.
- `Citation` remains the stable reference point.
- `Text` remains available for scriptures, translations, books, passages, and
  named textual entities already in the graph.
- `Term`, `School`, `Person`, and `Place` keep their existing meanings.

The new layer adds operational provenance nodes:

- `Corpus`: a bounded collection of materials selected for ingestion.
- `Source`: a provider, edition, dataset, repository, notebook, or collection
  from which documents are derived.
- `Document`: an ingestible unit such as a note, page, article, file, chapter,
  transcript, or extracted passage.
- `Evidence`: a claim-supporting excerpt, observation, annotation, or structured
  assertion derived from a document.
- `Work`: an abstract intellectual work that can group documents, editions,
  translations, or extracted documents.

## MVP Boundaries

This is architecture and data-model work only.

In scope:

- JSON seed data.
- Validation rules.
- Ontology and data-contract documentation.
- Deterministic graph artifacts generated by `make check`.

Out of scope for this step:

- FastAPI.
- Frontend changes.
- Embeddings.
- LLM extraction or answering.
- Vector storage.

## Minimum Pilot Shape

The pilot seed should include:

- One `Corpus` for the bounded pilot collection.
- One `Source` belonging to that corpus.
- One `Document` derived from the source.
- One `Work` when a document represents or belongs to a named intellectual work.
- Three `Evidence` nodes derived from the document.
- `Citation` nodes attached to the evidence.
- `EVIDENCES` links from evidence to existing `Concept` nodes.

## Relationship Semantics

- `BELONGS_TO_CORPUS`: places a source, document, work, citation, or evidence
  inside a corpus boundary.
- `DERIVED_FROM`: records provenance from a derived entity to its source entity.
- `HAS_DOCUMENT`: connects a corpus, source, or work to a document.
- `HAS_EVIDENCE`: connects a document, work, or citation to evidence extracted
  from it.
- `EVIDENCES`: connects evidence to the concept, term, text, or work it
  supports.
- `HAS_CITATION`: connects evidence, documents, works, or texts to a citation.

## Quality Rule

For MVP usefulness, an `Evidence` node should not stand alone. It should have:

- a `DERIVED_FROM` relationship to a `Document`;
- an `EVIDENCES` relationship to at least one concept or related knowledge node;
- a `HAS_CITATION` relationship to a `Citation` when a locator is available.

These quality expectations can become stricter validation rules after the pilot
data model settles.
